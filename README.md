# せかいラボ・ナビ（生徒用探究ナビWebアプリ）

生徒が「今やること」だけを見て探究を進められるナビゲーションアプリ。
玄関（種候補 → 問い → 章決め）から、実証・構成・行動・創作の各ルートの
論文・実践報告・制作ドキュメントまでを案内し、要所にAI壁打ちと思考ログの導線を置く。

## 設計の考え方（引き継ぐ人へ）

**アプリと生徒データを完全に分離している。** これが仕組みの核。

| 層 | 置き場所 | 中身 |
|---|---|---|
| アプリ（このフォルダ） | GitHub Pages（無料） | カード文言・AI壁打ちプロンプトだけ。**生徒の書き込みは一切保存・送信しない** |
| 生徒の一時メモ | 各自の端末のlocalStorage | 種候補・問い・チェック。端末の外に出ない |
| 思考ログ（本体） | Googleフォーム → 教員のスプレッドシート | 生徒の判断の記録。**閲覧は教員のみ**（フォームURLだけを生徒に配る） |

- 生徒が書いたものが他の生徒から見える場所は、どこにもない。
- アプリが消えてもログは残る。ログ側が壊れてもアプリは動く。
- 費用は0円（GitHub Pages無料枠・Google無料枠のみ。AIは生徒が学校で認められた各自のAIを使う）。

## 動かし方

### ローカルで試す
`index.html` をダブルクリックするだけで動く（ビルド不要・サーバー不要）。

### 公開する（GitHub Pages）
1. GitHubアカウントでリポジトリを作り、この `webapp/` の中身をアップロード（Web画面から「Add file → Upload files」でよい）
2. リポジトリの Settings → Pages → Branch: main / root を選んで Save
3. 数分で `https://ユーザー名.github.io/リポジトリ名/` が公開される
4. このURLを生徒に配る（Google ClassroomやQRで）

### ログフォームをつなぐ（生徒の二度書きをなくす設計）

フォームは**2問だけ**にする：①「章とカード」（記述式）②「ログ本文」（段落）。
アプリ内でログを書いて「📤 提出する」を押すと、**両方が自動で入った状態でフォームが開き、生徒は送信を押すだけ**。同時にマイログ（端末内の控え）にも自動保存されるので、同じものを二度書くことはない。

1. Googleフォームを上記2問で作る（回答はスプレッドシートへ。閲覧権限は教員のみ）
2. フォーム編集画面 → ︙ → 「事前入力したURLを取得」→ 両方の質問に仮入力してURLを発行 → URL中の `entry.XXXXXXX` を2つ控える
3. `js/config.js` の `LOG_FORM_URL`・`LOG_ENTRY_CARD`・`LOG_ENTRY_TEXT` に書き込む
4. フォーム未設定の間も、「提出する」は保存＋コピーとして動く（壊れない）

※項目別の詳細分析がしたくなったら、シート側でログ本文を関数分解するか、月1でAIに貼って集計する（`../docs/03_log_system_spec.md` 参照）。フォームを多項目にすると生徒の入力が二度手間になるので推奨しない。

## AI壁打ちの運用（Gemini）

生徒は学校のGoogleアカウントでGeminiを使う。方式は2つあり、混在できる。

- **基本（設定不要）**：壁打ち画面の「コピー」→ Geminiに貼って開始。
- **推奨（Gem方式）**：教員が各プロンプトを**Gem**に仕込んで共有リンクを `js/config.js` の `GEM_LINKS` に貼ると、生徒は「💎 Geminiで開く」のワンタップで開始できる。
  - 作り方：gemini.google.com → Gemを作成 → 「指示」欄に `data/prompts.js` の該当プロンプト全文を貼る → 学校ドメイン内で共有 → リンクをconfigへ。
  - 利点：生徒のコピペミスがなくなる／プロンプト改訂が教員側で一元化できる（アプリ更新不要）。
  - まず種発見（tane）と問い作成（toi）の2つだけGem化し、残りはコピペ運用から始めるのが現実的。
- Google Workspace for Education のGeminiは、入力内容がモデルの学習に使われない設定で提供される（生徒データの扱いの面でも個人アカウントより安全）。学校でのGemini有効化状況・年齢要件は管理者に確認すること。

## 内容を変えたいとき（プログラミング知識は不要）

文言はすべて3ファイルに分離してある。**この3つ以外は触らなくてよい。**

- `data/content.js` — カードの文言・ステップ構成（「今やること」「できたサイン」）
- `data/prompts.js` — AI壁打ちプロンプト全文
- `js/config.js` — フォームURL・マニュアル（スライド）のURL・GemリンクなどのURL設定

### 修正の手順（スマホでもできる）

1. github.com にログインし、このリポジトリを開く
2. 直したいファイル（例：`data/content.js`）をタップして開く
3. 右上の鉛筆マーク（Edit）をタップ
4. 文言を書き換える。**「"」や「,」や「`」などの記号は消さないこと**（文字の部分だけ変える）
5. 「Commit changes」を押す → 数分で本番URLに反映される

### 失敗したときの戻し方

リポジトリの「History」（時計マーク）から前の版を開き、「Revert」または前の内容をコピーして貼り直す。編集履歴は全部自動で残っているので、**壊して困る状態にはならない**。

## データの置き場所（プライバシーの整理）

| データ | 置き場所 | 見える人 |
|---|---|---|
| 提出された思考ログ | Googleフォーム → 教員のシート | 教員のみ |
| マイログ（生徒の見返し用） | 生徒各自の端末のlocalStorage | 本人のみ。送信されない。端末を変えると消える |
| 種候補・問い・チェック | 同上（端末内） | 本人のみ |
| アプリの文言・プロンプト | このリポジトリ（公開） | 誰でも（生徒データは含まれない） |

## 設計上の約束（変えないでほしいこと）

1. **アプリに生徒データを保存する機能を足さない**（コメント欄・共有機能など）。線引きが崩れる
2. **語彙ルール**：実証＝仮説／構成・行動＝見立て／創作＝ねらい。「仮説」で統一しない
3. **行動・創作を研究に寄せない**：検定・統計を必須にしない。合格ラインは「前後の差が感想以外の痕跡1つ以上」
4. **AI壁打ちの4条件**（`../docs/02_ai_kabeuchi_spec.md`）：生徒が先に出すまで提案しない／序盤は問い返しのみ／案は引用形式のみ／最後の判断と理由は生徒が書く
5. **個人情報**：本名・逐語をログやプロンプトに入れさせない
